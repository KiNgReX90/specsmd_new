SPECSMD · flow reference

What each flow writes, and how it builds.

Five installable flows ship in one package. They diverge on two axes: the documents they leave behind, and how work is decomposed and executed. This is a schematic map of both — plus a straight answer on Turbo-A, which is not a flow at all.

01 — orientation

The five flows at a glance

One row per flow. The split that matters most: AI-DLC is a deep, checkpoint-gated lifecycle; INFERNO is a flat, autonomous, parallel build; FIRE sits between them; IDEATION is pre-build; SIMPLE is the minimum.

Output documents & lifecycle, flow by flow

Left column = the files left on disk (every path is literal). Right column = the order work moves through. Amber chips are human approval gates; accent-bordered chips run in parallel.

AI-DLC heavyweight · governed memory-bank/

The deepest flow: a five-level decomposition driven by four agents (Master, Inception, Construction, Operations) across three sequential phases, with a human checkpoint at every DDD stage. Documentation comes before code — model and design stages write no code at all.

Sequential & gated. Stages run strictly in order; the agent stops for human approval at each boundary. Bolts within a unit run serially; bolts in different units may overlap if no dependency links them.

FIRE adaptive · runs .specs-fire/

A middle path: three agents (orchestrator, planner, builder) and a flatter Intent ▸ Work-item ▸ Run hierarchy. Each work item's complexity picks an execution mode that decides how many checkpoints it gets. A design doc is written only for high-complexity items.

Output documents

│   ├ tech-stack.md · coding-standards.md
│   └ testing-standards.md · system-architecture.md

│   ├ brief.md
│   └ work-items/
│       ├ <w>.md          # AC, complexity, mode
│       └ <w>-design.md   # high complexity only
└ runs/<run>/

    ├ test-report.md
    ├ review-report.md

Lifecycle
Plan
capture intent→decompose work-items→design (high)
Run · per mode

Mode-adaptive gates. autopilot = 0 gates · confirm = 1 (approve plan) · validate = 2 (design + plan). A run covers one item (single), a group (batch), or all compatible items (wide) — you choose the scope.

INFERNO autonomous · parallel .specs-inferno/

The flat, autonomous build flow. The planner decomposes an intent straight into work items — no stories, no bolts — each carrying its own dependency list and file ownership in frontmatter. The orchestrator then runs builders in parallel and merges the result itself.

Output documents ~11 files · ~661 lines*

│                       #   work-items, runs, claims
├ config.yaml           # optional: models, autonomy,
│                       #   verify cmds, halt, delivery
├ intents/<id>/
│   ├ brief.md
│   └ work-items/
│       ├ <w>.md         # frontmatter: depends_on,
│       │                #   ownership.editable, kind,
│       │                #   complexity, context.{required,
│       │                #   patterns,tests}; body: AC
│       └ <w>-design.md  # optional, high complexity
└ halt-notes/           # budget-cap halt only
    ├ _intent-<id>.md
    └ <w>.md             # per-item resume checkpoint
Lifecycle
Plan (planner) — stops
capture intent→decompose→‖ scribes write→review?
Build (orchestrator)
claim intent→1 worktree→‖ ready frontier→verify+commit→merge→teardown

Parallel by ownership, not by worktree. All builders share one intent worktree. Items with disjoint ownership.editable run concurrently; items that share a file — or are linked by depends_on — serialize. The orchestrator integrates results one at a time, re-computing the runnable frontier after each.

IDEATION creative · pre-build .specs-ideation/

Not a build flow at all — it produces thinking, not code. A single orchestrator runs three skills over a resumable session: generate ideas, evaluate them, then shape the survivors into shareable concept briefs.

Output documents 4 per session
.specs-ideation/sessions/<id>/
├ session.yaml          # phase, favorites, scores
├ spark-bank.md         # idea batches (S1-1, S1-2…)
├ flame-report.md       # Six-Hats eval + ranked shortlist
└ concept-briefs/
    └ <concept>.md       # polished, pitch-ready brief
Lifecycle
Spark → Flame → Forge
spark · generate→flame · Six Hats + score→forge · Dream/Reality/Critic

Generate → evaluate → shape. Spark forces cross-domain diversity; Flame scores ideas on impact × feasibility; Forge runs Disney-strategy passes into a concept brief. Each phase is resumable from session.yaml, and the three skills also run standalone.

SIMPLE minimal · flat specs/<feature>/

The smallest flow: one agent, three markdown files per feature, no nesting and no worktrees. State is just which files exist. Each phase is gated by an explicit "approved" before the next is written.

Output documents 3, flat
specs/<feature>/
├ requirements.md       # EARS-format acceptance criteria
├ design.md             # architecture + Mermaid diagrams
└ tasks.md              # numbered checkbox task list
Lifecycle
Three gated phases, then execute
requirements→approve→design→approve→tasks→approve→execute 1 task

One phase, then one task, at a time. The agent drafts first and waits for explicit approval before advancing a phase; in execution it implements a single task, ticks its checkbox, and stops for review.

⚒ correcting one detail

You pictured INFERNO running “a separate worktree, and under that sub-worktrees.” It's actually simpler: there is exactly one worktree per intent, and all builders run inside it.

Parallelism doesn't come from nested worktrees — it comes from file ownership. Every work item declares ownership.editable; the orchestrator dispatches any items whose ownership sets don't overlap (and whose depends_on is satisfied) at the same time. Two items that touch the same file simply serialize. That's why independent items build concurrently without edit collisions, while coupled work naturally falls into order.

04 — the one that isn't a flow

So what is Turbo-A?

Short answer: not a flow. Turbo-A and AI-DLC-light are eval arms — candidate document shapes from the doc-slimming experiment, not things you install. The five flows above are the product; the arms are an A/B test of how lean AI-DLC's paperwork can get without losing build quality.

your hunch — mostly right

You thought Turbo-A was a combination of INFERNO and AI-DLC. That's close. Turbo-A keeps AI-DLC's inception decisions (its quality edge — trimmed requirements + a decisions ledger), but collapses the document layers so the story and the work-item become a single full-fidelity file per buildable thing — acceptance criteria, edges and caveats in the body, dependencies and ownership in the frontmatter (the INFERNO work-item shape) — and it builds with INFERNO's parallel TDD builders.

So it genuinely fuses AI-DLC's thinking with INFERNO's work-item shape and parallel execution — but as one slimmed AI-DLC doc shape, not a runtime that bolts the two flows together.

Their document structure & execution

The same treatment as the five flows — that's the whole point of the experiment. The executor is held constant: every arm's docs are handed to the same canonical INFERNO TDD-gated builder on the same frozen base. Only the document shape below changes.

The verdict — 3 runs, 2 apps, ranking held

A slim shape won every run; the heavy AI-DLC control was Pareto-dominated and the lean INFERNO control failed the caveat gate each time. Two shapes are worth adopting:

The most economical arm and the strongest code on BTW and financial-middleware. More sensitive to per-build idiom slips (fixable with a lint gate) — pick it when reviewer time is the bottleneck.

safest default

strong on every run, never below 8.06

build quality 8.06–8.55 / 10 · docs −73 to −81%

The only arm strong across all three runs — outright won procesbeheer and matched the heavy control on the others. Pick it when one shape has to be safe across many intents and apps.

The heavy control bought nothing

AI-DLC (full, ~3,454 lines) was Pareto-dominated on runs #2 and #3 — its doc-bloat purchased neither correctness nor fidelity.

The lean control is dangerous

INFERNO (bare, ~661 lines) dropped load-bearing caveats behind unresolved links and failed the caveat gate on all three runs — once producing a hollow build (3.07 / 10).

The win is structural, not one shape

Which slim shape leads flips by app (turbo-a vs aidlc-turbo), but a slim shape wins every time: write each load-bearing fact once in the builder's doc and link — don't restate, and don't hide it behind a hop.
