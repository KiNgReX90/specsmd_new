specsmd · the doc-slimming experiment

AI-DLC · AI-DLC Turbo · INFERNO

How much documentation does an AI agent actually need to build good code? We measured it — same code change, different documentation shapes, blind-judged. This is what the experiment showed, and the three flows it settles on.

AI-DLC — the full methodology, heaviest docs AI-DLC Turbo — same phases, slim docs INFERNO — autonomous, parallel
What we did

The experiment

We had the same code change built from several different documentation shapes (flows) — from the full, heavy AI-DLC documentation down to sharply slimmed variants. Under the hood the build process was identical everywhere; only the form and amount of documentation changed. That isolates exactly one variable: does more documentation help at all?

How it was judged

Outcomes were blind-judged by a panel of three independent reviewers on a weighted 0–10 rubric (correctness, completeness, fidelity to the requirements, maintainability, and more).

We repeated the experiment across three changes of increasing difficulty, so we could see whether the ranking holds. No single score counts — the question is whether the same flow keeps winning as the work gets harder.

The three flows

What each flow does

AI-DLC
the full methodology

AI-DLC is the full methodology with the heaviest documentation — it's the documentation that's heavy, not the process. Its inception phase produces an extensive set of documents: a domain model, detailed technical design, architecture decision records (ADRs), and a fine-grained breakdown into stories and bolts with full traceability. Lots of documentation, lots of mandatory paperwork, with facts written out again in several places. In the experiment, all that extra documentation bought no quality gain.

AI-DLC Turbo
same process, slim docs

AI-DLC Turbo runs the exact same phases as AI-DLC — the inception itself is not shortened. What gets lighter is the documentation, not the process: no mandatory paperwork, no repetition. Inception produces only requirements, one decisions-and-gates ledger, and stories (no bolts). Then story-driven construction with human checkpoints. Same process, same steering, a fraction of the documentation.

INFERNO
autonomous · parallel

INFERNO is the autonomous, parallel build flow. It decomposes an intent automatically into work items and lets parallel builders build them at the same time in one worktree. Coupled work serializes on its own via dependencies; independent work builds concurrently, without the builders stepping on each other. This is the shape that scored the highest build quality in the experiment.

Configure first

INFERNO starts with configuration

INFERNO opens with a Config phase. On install the flow looks for its configuration file; if it's missing, you're guided to the configuration wizard. What you set here decides how the rest of the flow behaves.

PHASE 0 · CONFIG — read before anything runs
→On install, look for the config file; if it's absent, the wizard walks you through it.
→What you configure:
model tiers — strong vs cheap autonomy level finalize verification — build / test / lint gate delivery mode optional: budget-cap halt optional: knowledge index

The crucial one is the delivery mode. It picks between two ways the work lands:

delivery.mode = auto-close

INFERNO verifies the intent worktree itself against your gates, merges it into your main branch, and pushes. Fully hands-off — no human in between.

delivery.mode = merge-request

INFERNO stops after verification and opens a merge request (or leaves the branch for review). You review everything before it lands.

In your repository

What gets created in Git

A common question: what does each flow actually create in your Git repository — branches, worktrees, merge/pull requests? And does it happen per work item, or only per intent? The answer differs sharply between the human-in-the-loop flows and INFERNO, and within INFERNO between its two delivery modes.

First, one thing to untangle. INFERNO has two separate settings that are easy to confuse. The autonomy level (full / review) only controls whether the planner pauses for your sign-off after planning and before the build — it says nothing about Git. What gets created in Git is decided entirely by the delivery mode: auto-close or merge-request.
  AI-DLC & AI-DLC Turbo INFERNO — auto-close INFERNO — merge request
Who manages Git? You. The flow doesn't touch Git. The orchestrator, fully autonomous. The orchestrator, with you as the final gatekeeper.
Worktree None — you work in your normal checkout. One per intent (inferno-intent/<id>), where all builders work in parallel. Same — one per intent.
Branch per intent None; you decide your own branches. One intent branch, in that worktree. One intent branch, in that worktree.
Per work item / story Nothing. The flow writes code and documents into your working tree; committing and branching stays yours, as always. No separate branch or MR — each builder's result is committed directly onto the intent branch. Yes: each work item gets its own branch (inferno-item/<intent>/<item>) and a non-blocking merge request into the intent branch (for traceability / auto-merge).
At finalize You commit and merge in your own workflow. The intent branch is merged locally into the base branch, the worktree is torn down, and the base branch is pushed. The intent branch is pushed and one merge request is opened into the base branch. The orchestrator stops there; the worktree stays for review.
Merge / pull request Only if you open one yourself. None. One per work item (non-blocking) + one into the base branch (the review gate). Works with GitLab (merge request) and GitHub (pull request).
Review gate Your normal review, outside the flow. None — the code lands with no one in between. That one merge request into the base branch. Nothing lands before you approve it.

In short: AI-DLC and AI-DLC Turbo create nothing in Git themselves — you branch and commit as usual. INFERNO always works in one worktree with one branch per intent, never per work item. Only the merge-request mode also creates something per work item (its own branch plus a non-blocking merge request); in auto-close everything stays on that single intent branch and INFERNO merges it itself.

In one line each

Which to reach for

AI-DLC

The full methodology with the heaviest documentation — kept as the baseline the experiment measured against. The extra paperwork didn't buy quality.

AI-DLC Turbo

Same process and same steering as AI-DLC, a fraction of the documentation. The most consistent all-rounder and the safest default to lock in.

INFERNO

Highest build quality, autonomous and parallel in one worktree — delivering either by self-merge or by opening a merge request, whichever the config says.

🔥
The win is structural, not a specific tool. Write each load-bearing fact once and link to it — rather than restate it (heavy) or defer it behind an unresolved hop (too lean). That's what a slim shape gets right and the heavy baseline never does.
