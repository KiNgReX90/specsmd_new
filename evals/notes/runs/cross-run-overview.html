Cross-run synthese · run #1–#3 · 2 apps · 2 stacks · 1 methode

Drie runs, twee apps — en de ranking houdt stand. Documentbloat kan 73–91% slanker zonder bouwkwaliteit te verliezen.

Dezelfde eval draaide nu drie keer met identieke methode — zelfde 7 documentvormen, dezelfde vastgezette TDD-bouwer, dezelfde gewogen rubriek, dezelfde twee harde gates. Alleen de app en het intent varieerden. Elke run wijst dezelfde kant op: de zware AI-DLC-controle wordt Pareto-gedomineerd, de lean pointer-only-controle is steeds de slechtste build én zakt voor de gate, en een slanke vorm wint élke keer.

02 · De aanbeveling

Twee vormen om te adopteren

De topvorm wisselt per app, dus de keuze hangt af van wat je vastlegt: één vorm die over intents én apps heen veilig is, of de leanste vorm waar reviewlast bindt.

AANBEVOLEN
Veiligste cross-intent default
aidlc-turbo
jurypanel 8,06 – 8,55/10 élke run · docs −73 tot −81%

De enige arm die over álle drie de runs sterk is — nooit onder 8,06. Won procesbeheer ronduit (Pareto-domineert de zware controle bij −73% docs) en evenaart de zware controle op de andere twee. Kies dit als één vorm over intents en apps heen moet werken.

Laagste reviewlast · beste code waar het leidt
turbo-a
won BTW (8,65) & fmw (8,58) · docs tot −91%

De leanste vorm én de beste build op twee van de drie runs. Gevoeliger voor losse per-build slips (een polling source-mutatie, een @Transactional-self-invocation) — fixbare idioom-gates, geen vormgebreken. Kies dit waar reviewlast bindt.

De twee controles verliezen consequent

De zware AI-DLC-controle (0 economie) wordt in run #2 én #3 Pareto-gedomineerd — de doc-bloat kocht geen correctheid. De lean pointer-only-controle (max economie) zakt voor de gate in álle drie de runs en bouwde op fmw een ronduit holle engine (3,07).

Wat kies je?

Eén vorm vastleggen → aidlc-turbo. Reviewlast bindt → turbo-a. Beide snoeien 73–91% van de docs en matchen of verslaan de zware controle op bouwkwaliteit.

05 · Wat over runs heen stabiel is

De cross-run conclusie

Vier bevindingen die in alle drie de runs gelden — de kern van waarom de stelling generaliseert buiten ViewPoint.

Lean pointer-only controle verliest en zakt voor de gate — alle drie de runs

Caveats achter onopgeloste links weglaten levert echte, load-bearing build-defecten; op fmw zelfs een holle build (groene tests, nul veld-mappings, 3,07). De goedkope, build-loze Stage-A-gate voorspelde het elke keer, vóór er een regel gebouwd was.

Slank ≠ slechter — cross-app bevestigd

Een 73–91% slankere documentvorm evenaart of verslaat de zware AI-DLC-controle op bouwkwaliteit. Die controle is nu in run #2 én #3 Pareto-gedomineerd; de doc-bloat kocht correctheid noch getrouwheid — op fmw-tracking liet de zware controle zelfs correcties vallen die de slanke armen wél bouwden.

De winst is structureel, geen specifieke vorm

De topvorm wisselt per app — turbo-a (BTW & fmw) vs aidlc-turbo (procesbeheer) — maar een slanke vorm wint élke run en de zware controle nooit. Schrijf elk load-bearing feit één keer in de bouwer-doc en link, i.p.v. herhalen (zwaar) of achter een onopgeloste hop verstoppen (lean).

Documentvorm discrimineert waar de logica het zwaarst is

Op service/engine/tracking-units waaieren de armen ver uit en is de volgorde stabiel; op dun delegatie-/UI-werk (en MED-complexiteit polling) is het ruis — precies waar de twee echte per-build slips landden.

Aanbeveling & resterende gate

Adopteer een slanke vorm: aidlc-turbo als de veilige cross-intent default; turbo-a waar reviewlast bindt. Beide snoeien 73–91% van de docs zonder bouwkwaliteit te verliezen.

Vóór adoptie in de flow: vouw de twee gates in die run #2/#3 aanwezen — een idioom/lint-gate (geen @Transactional-self-invocation, geen source-mutatie in een read-only laag, geen cross-layer enum-plaatsing) en een caveat-inline-check — zodat geen arm herhaalt wat de panels signaleerden.
